PRÁCTICA 3 — Vulnerabilidades XSS

1. Reflected XSS (vulnerable)

Lo que escribas aquí se inserta usando innerHTML

2. Stored XSS (vulnerable con localStorage)

Añade una "nota". Se guarda en localStorage y luego se renderiza.

Notas guardadas:

Solución en este caso concreto

Si en lugar de innerHTML usamos textContent, el payload se muestra como texto plano y **no se ejecuta**.

Resultado seguro:

Pistas de prueba (recordatorio)

1. Reflected XSS (Primera caja roja)

Qué escribir: Hola Deusto <img src=x onerror=alert('Ataque_XSS')>
Qué debería pasar: aparecerá el texto "Hola Deusto" y se abrirá un alert().
Por qué pasa: al usar innerHTML, el navegador interpreta el input como código HTML.

2. Stored XSS (Segunda caja roja)

Qué escribir: <details open ontoggle=alert('Nota_Maliciosa')>Click para ver nota</details>
Qué debería pasar: al guardar la nota se ejecutará el alert, y volverá a aparecer al recargar.
Por qué pasa: la nota se guarda en localStorage y se renderiza con innerHTML.

3. Solución (Caja verde)

Qué escribir: <img src=x onerror=alert('XSS')>
Qué debería pasar: no ocurre nada, solo se muestra el texto.
Por qué pasa: aquí usamos textContent, que trata la entrada como texto plano.

Resumen:
Vulnerabilidad: insertar datos del usuario con innerHTML.
Objetivo: lograr un alert() mediante XSS.
Arreglo: usar textContent.
